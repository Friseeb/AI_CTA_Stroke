1. Classification Pipeline Overview

Each patient folder may contain multiple Export_* subdirectories. The pipeline processes them in two phases:

Phase 1 — Select Best Export
Classify every Export_* folder. When multiple exports match the same scan type, keep the one with the most source DICOM files (most complete data).
→
Phase 2 — Convert Winner
Convert only the winning export for each scan type to NIfTI. Apply size gate. Write sidecar JSON + manifest row.

Each export folder is classified through 4 sequential steps:

Step 1 — Coverage Gate
Measure anatomical coverage from ImagePositionPatient z-coordinates (max across all series).
If outside the bounds → disqualify (score = 0).
→
Step 2 — Study-Desc Veto
Per-type: if any exclude_study_desc_keywords substring appears in StudyDescription → disqualify (score = 0).
Currently only ThoraxCT uses this.
→
Step 3 — Keyword Scoring
Match DICOM text fields against keywords.
StudyDescription +150 · ProtocolName +80 · BodyPart +50.
Minimum ≥ 50 pts required.
→
Step 4 — Winner
Highest-scoring type wins.
If no type qualifies → export is UNCLASSIFIED (shown as error in report).

2. Step 1 — Anatomical Coverage Exclusion (Hard Gate)

coverage_mm = | z_max − z_min | — measured from the ImagePositionPatient DICOM tag, sampling ~20 evenly-spaced files from each eligible series in the export, then taking the maximum z-spread across all series.

This metric is scanner-independent and center-independent — it reflects true anatomical extent regardless of PACS naming conventions or language.

If ImagePositionPatient is absent (tag missing in all series), the coverage gate is skipped and classification falls back to keywords only.
Why max across all series? — Exports often include sagittal and coronal reformats alongside the axial source acquisition. Sagittal slices vary in x (left–right), not z (superior–inferior), so their z-spread is ≈ 0 mm. If the largest-by-file-count series is a sagittal reformat, coverage would be falsely reported as ~0 mm, disqualifying every scan type and marking the export as UNCLASSIFIED. Taking the maximum across all series ensures the axial acquisition (which has the real head-to-foot coverage) drives the estimate.

Overlap zones — multiple types may survive the coverage gate simultaneously (e.g. a 600 mm scan passes both ThoraxCT and TotalBodyCT). The study-description veto (Step 2) and keyword scoring (Step 3) resolve the tie. For ThoraxCT specifically: even though its z_max is now 750 mm, any study whose description contains "abdomen" or "pelvis" is vetoed before scoring — so combined CAP studies still route to TotalBodyCT or AbdomenCT.

3. Step 2 — Study-Description Veto

After the coverage gate, each scan type can optionally declare exclude_study_desc_keywords — a list of substrings that, if found in StudyDescription, force the type's score to 0 (disqualify it entirely).

This is a hard veto: it fires before keyword scoring and cannot be overcome by a high keyword score.

Currently only ThoraxCT uses this field: exclude_study_desc_keywords = ("abdomen", "pelvis").

Rationale: ThoraxCT's z_max was raised to 750 mm and the keyword list was expanded (adding "chest", "cardiac cap"). Without the veto, combined studies like "CT Thorax/Abdomen/Pelvis" (z ≈ 650 mm, contains "chest") could score +150 for ThoraxCT and compete with TotalBodyCT. The veto eliminates this risk cleanly. "CT Cardiac CAP" is unaffected — neither "abdomen" nor "pelvis" appears in that study description.

4. Step 3 — Keyword Scoring

DICOM FieldPointsNotes
StudyDescription+150Most reliable label — highest weight. Only one keyword needs to match.
ProtocolName+80Medium weight.
BodyPartExamined+50Lowest weight. Sometimes absent or generic.

All keyword matching is case-insensitive substring search. At most one bonus per field per type (first match wins, loop breaks).
Minimum threshold: ≥ 50 pts — a type with no keyword match at all is rejected even if it passed the coverage gate.

7. Other Pipeline Behaviours

FeatureBehaviour
Parallel processing --workers N controls the number of parallel patient processes (default: max(1, cpu_count // 2)). Uses ProcessPoolExecutor — subjects are processed in parallel, each worker handles one patient at a time.
Best-export selection Phase 1: all Export_* folders are classified first. When multiple folders match the same type, the one with the highest total DICOM file count (most source data) is selected. Duplicates are logged as [DUPLICATE].
Idempotent re-runs If the output NIfTI already exists → status = skip_exists. No re-conversion.
Multi-phase export CardiacCT, TotalBodyCT, AbdomenCT: all source series exported as _ph00, _ph01, …
Slice ordering Sorted by z-position (ImagePositionPatient) if ≥70% of slices have the tag, otherwise by InstanceNumber.
Dimension deduplication Mixed-size series (axial + coronal/sagittal reformats under the same SeriesInstanceUID) filtered to dominant Rows×Columns pair before stacking.
Coverage sampler Reads ~20 files per eligible series; returns the maximum z-spread across all series. Prevents SAG/COR reformats (z-spread ≈ 0) from falsely disqualifying the export.
Subject sort order Numeric sort (1, 2, 3 … 299), not alphabetic (1, 10, 100 …).
Output report HTML report (conversion_report_<timestamp>.html) + CSV manifest. Error badges show reason codes: convert (exception during SimpleITK), unclassified (DICOM present but no scan type matched), no DICOM (Export folder is empty). Unclassified and empty exports shown in the Exports column.
Batch log Full console output saved to batch_log_<timestamp>.txt in the output root.
